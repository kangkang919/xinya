本仓库采用「纯 .env + process.env」的轻量级配置方案，没有引入专门的配置库（如 dotenv、config、conf），所有运行时参数通过 Node.js 原生 process.env 在模块加载时读取。

## 1. 配置来源与环境
- 开发模板：.env.example 提供必需变量清单（未包含在本仓库中，但由 .gitignore 保护）。
- 生产备份：doc/.env.production.备份 保存了线上环境的完整变量快照，含数据库连接、JWT 密钥、SMTP 凭据等。
- 部署脚本：deploy.sh 在服务器 /www/wwwroot/xinya 下执行安装、迁移、构建与 PM2 启动；PM2 进程定义见 ecosystem.config.js，仅设置 NODE_ENV=production，其余变量依赖系统级 .env 或宿主环境。

## 2. 核心环境变量清单
- DATABASE_URL: Prisma PostgreSQL 连接串，必填。
- JWT_SECRET: JWT 签名密钥，开发回退为 xinya-dev-secret-change-in-prod。
- SMTP_USER/SMTP_PASS: QQ 邮箱 SMTP 认证，必填。
- NEXT_PUBLIC_BASE_URL: 前端公开基础 URL，用于邮件链接拼接。
- NEXT_PUBLIC_APP_URL: Magic Link 目标站点，部分路由使用此变量。
- DEEPSEEK_API_KEY: AI 题目生成 API Key，可选。
- SILICONFLOW_API_KEY: 硅基流动 API Key，可选。
- NODE_ENV: 运行环境，development 或 production。

## 3. 配置加载位置与模式
- lib/auth.ts: 集中导出 COOKIE_CONFIG（cookie 名称、安全选项、过期时间）和 JWT 密钥，供所有认证相关 API 路由复用。
- lib/prisma.ts: 根据 NODE_ENV 切换 Prisma 日志级别（开发输出 query/error/warn，生产仅 error）。
- lib/mailer.ts: 模块顶层直接创建 nodemailer transporter，依赖 SMTP_USER/SMTP_PASS。
- 各 API route: 按需从 process.env 读取 NEXT_PUBLIC_* 前缀变量，用于构造外部链接。

## 4. 架构约定
- 无统一 config 模块：每个需要配置的子模块自行 import process.env.XXX，缺乏集中校验与类型约束。
- 敏感信息不入库：.env 被 .gitignore 排除，生产凭据以独立备份文件形式存放在 doc/ 目录。
- Next.js 配置极简：next.config.ts 为空壳，未启用任何运行时注入的 Next 配置项。
- PM2 只管理进程：ecosystem.config.js 仅声明应用名、端口、实例数与 NODE_ENV，不承载业务配置。

## 5. 开发者应遵循的规则
- 新增环境变量：同步更新 .env.example 并在所有引用处提供合理的默认值或明确报错。
- 禁止硬编码密钥：JWT 密钥、数据库密码、API Key 一律走 process.env，不得写死在源码中。
- 区分 NEXT_PUBLIC_* 与非公开变量：只有以 NEXT_PUBLIC_ 前缀的变量才会被打包到浏览器端，其他变量仅限服务端使用。
- 生产部署：确保服务器上的 .env 与 doc/.env.production.备份 保持一致，并通过 deploy.sh 一键完成迁移与重启。
